1. Изучи, проанализируй пакет, название которого я указал рядом с командой. Пакет находится в `./packages/%package_name%`. Все дальнейшие обсуждения касаются только директории этого пакета.
2. Прочитай целиком существующий `./packages/%package_name%/README.MD`. Вся дальнейшие изменения будут происходить только в этом файле.
3. Важно! Работать нужно только с контентом до `[//]: DOCUMENTATION_SECTION_START`. Весь контент файла `README.MD` от `[//]: DOCUMENTATION_SECTION_START` до `[//]: DOCUMENTATION_SECTION_END` генерируется автоматически и доработок не требует.
4. Изучи story пакета в папке `./packages/%package_name%/stories/**.story.tsx`. Изучи доступные контролы, основные компоненты и пропы в `./packages/%package_name%/src/`, e2e тесты из директории `./packages/%package_name%/__test__/**.spec.ts`. Если что-то непонятно, не хватает каких-то полезных данных (предназначение, цель, смысл), и нужны уточнения – задай мне вопросы-уточнения.
5. Сформулируй описание главного компонента/компонентов, входящих в пакет `@snack-uikit/%package_name%` в секции `Description`, исходя из количества компонентов:
   - Если файл(ов) `./packages/%package_name%/stories/**.story.tsx` **ровно один**, то внутри пакета есть один основной компонент. В этом случае:
     - используй **одну** общую секцию второго уровня `## Description` (она относится ко всему пакету и к этому компоненту одновременно);
     - для примера используй **одну** секцию второго уровня `## Example`.
   - Если файлов `./packages/%package_name%/stories/**.story.tsx` **несколько**, то внутри пакета несколько основных компонентов. В этом случае:
     - создай **одну** общую секцию второго уровня `## Description` с описанием общих моментов для всего пакета;
     - для каждого основного компонента создай заголовок второго уровня `## ComponentName` (в качестве значения – имя описываемого компонента);
     - внутри каждого `## ComponentName` создай секции третьего уровня `### Description` и `### Example`, относящиеся к этому компоненту;
     - добавь отличительные особенности по каждому компоненту. Старайся не повторять одинаковые вещи в описаниях (всё общее должно быть в `## Description` выше, относящемуся ко всему пакету). 
   - Как понять, что компонентов несколько внутри пакета? Должно быть несколько папок `./packages/%package_name%/src/components/%SomeComponent%` и несколько одноимённых `./packages/%package_name%/stories/%SomeComponent%.story.tsx`.
   - При описании всех `Description` используй ненумернованный список, где применимо. Сохраняй стиль написания списка как в компонентах `./packages/accordion/README.MD` и `./packages/alert/README.MD`. В описании явно укажи, что представляют собой компоненты и чем отличаются друг от друга.
   - Если название компонента отличается от названия файла (`./packages/%package_name%/stories/%SomeComponent%.story.tsx`), подсвети это в описании и используй фактическое название компонента на основании кода вызова в story.
   - Укажи отдельным ненумернованным пунктом ссылку на Figma-макет данного компонента. Ссылка хранится в `./packages/%package_name%/stories/%SomeComponent%.story.tsx` в самом низу файла, в экспортируемом объекте в `parameters.design.url` (`params.design.name` будет иметь значение Figma). Укажи это в формате: Figma: %имя_компонента_обёрнутое_в_ссылку%.
6. Для каждого из основных компонентов внутри пакета укажи свой пример в соответствующей секции `Example` файла `./packages/%package_name%/README.MD`:
   - если компонент один (один story-файл), используй единственную секцию `## Example`;
   - если компонентов несколько (несколько story-файлов), добавляй примеры в соответствующие секции `### Example` внутри блоков `## ComponentName`.
7. Не удаляй в `./packages/%package_name%/README.MD`:
   - Другие секции и/или иную полезную информацию (при наличии);
   - Заголовок первого уровня (название компонента, например `# ComponentName`).
